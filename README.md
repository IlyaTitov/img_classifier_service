# Сервис классификации изображений (YOLO)

Веб-приложение для загрузки фотографий, **асинхронного** детектирования объектов моделью **YOLOv8** (Ultralytics), просмотра результатов и архива по пользователю.

## Pipeline

1. Пользователь регистрируется или входит; API выдаёт JWT.
2. Загрузка изображения сохраняет файл на диск, создаёт запись в **PostgreSQL** и ставит задачу в **Celery** (брокер **Redis**).
3. Воркер Celery запускает **YOLO**, сохраняет координаты рамок и уверенность в БД, сохраняет размеченное изображение (`processed_…`) и обновляет денормализованный счётчик `detection_count`.
4. Клиент опрашивает `GET /v1/image/{id}/status` (лёгкий endpoint без детекций), пока `processing_complete` не станет `true`, затем запрашивает полный результат через `GET /v1/image/{id}`.
5. Архив поддерживает фильтры по датам, сортировку и постраничную выдачу (`limit`/`offset`).

## Стек

| Компонент | Технология                        |
| --------- | --------------------------------- |
| API       | FastAPI, Uvicorn, JWT             |
| Фронтенд  | React (Vite), Nginx               |
| Очередь   | Celery + Redis                    |
| Кэш       | Redis (отдельная БД `/1`)         |

| Хранилище | PostgreSQL, файлы на volume       |

## Быстрый старт (Docker Compose)

1. Скопируйте переменные окружения:

   ```bash
   copy .env.example .env
   ```

2. Положите веса модели (например `yolov8s.pt`) в каталог, который монтируется в контейнер как volume **`weights`** (см. `.env.example`: путь `YOLO_MODEL_PATH=/app/weights/yolov8s.pt`).

3. Запуск:

   ```bash
   docker compose up --build
   ```

4. В браузере: фронтенд обычно на **http://localhost** , API напрямую — **http://localhost:8000**.

Нагрузочное тестирование (профиль `loadtest`, UI Locust на порту **8089**):

```bash
docker compose --profile loadtest up locust
```

## HTTP API (ключевые маршруты)

| Метод | Путь                    | Описание                                                                                      |
| ----- | ----------------------- | --------------------------------------------------------------------------------------------- |
| POST  | `/v1/auth/register`     | Регистрация (`login`, `password`)                                                             |
| POST  | `/v1/auth/login`        | Выдача JWT (`access_token`)                                                                   |
| POST  | `/v1/image/upload`      | Загрузка файла; возвращает `id` и `task_id`                                                   |
| GET   | `/v1/image/`            | Архив пользователя (`limit`, `offset`, `sort_by`, `order`, фильтры по дате)                   |
| GET   | `/v1/image/{id}/status` | **Лёгкий статус** — только `processing_complete`, `task_id`, `retry_after`; без JOIN детекций |
| GET   | `/v1/image/{id}`        | Полный ответ с координатами всех детекций (кэшируется в Redis)                                |

Для защищённых методов: заголовок `Authorization: Bearer <token>`.

| `backend/`   | FastAPI, Celery-задачи, YOLO, модели SQLAlchemy               |

## Архитектура и поток данных

```
Браузер → Nginx → FastAPI (Uvicorn)
                      │
                      ├── PostgreSQL  (записи, архив)
                      ├── Redis /1    (HTTP-кэш, TTL 3–60 с)
                      └── Redis /0 → Celery worker → YOLO → PostgreSQL / Disk
```

**Разделение ответственности:**

- FastAPI остаётся лёгким: приём файлов, CRUD, JWT. Тяжёлая работа не блокирует event loop.
- Celery изолирует инференс и ограничивает параллелизм (`--concurrency`).
- Redis `/1` поглощает повторные запросы к одному `image_id` во время поллинга.

**Безопасность :** пароли хешируются (bcrypt), доступ к изображениям привязан к `user_id`.

## Целевое ТЗ по производительности

Требования задаются для профиля нагрузки **100 одновременных пользователей**, появляющихся с интенсивностью **10 пользователей в секунду** .
| Показатель | Требование |
|------------|------------|
| **Пропускная способность** | Суммарная нагрузка на API **не менее 290–295 RPS** (агрегированный показатель по всем типам запросов в тесте). |
| **Эффективность поллинга** | На эндпоинтах **проверки статуса изображения** (`GET …/status` и/или полный опрос `GET …/{id}` в сценарии опроса) суммарный поток **не ниже 260 RPS** при работе кэша Redis на полном ответе изображения. |
| **Надёжность** | **Success Rate не ниже 99,9%** при указанном числе одновременных пользователей |

## Целевая нагрузка и производительность

Конфигурация по умолчанию (1 Uvicorn-воркер, Celery `--concurrency=2`, PostgreSQL и Redis в том же Compose) **намеренно сбалансирована** под следующий профиль нагрузки:

| Показатель                    | Целевое значение               | Обоснование                                                                               |
| ----------------------------- | ------------------------------ | ----------------------------------------------------------------------------------------- |
| Одновременные пользователи    | **~100**                       | Подтверждено нагрузочным тестом Locust (100 users, ramp 10/s): 0 ошибок, медиана < 500 мс |
| Суммарный RPS                 | **~295 RPS**                   | Измерено: Aggregated median 1000 мс при ~294 Current RPS                                  |
| Поллинг статуса `GET /status` | **~260 RPS** (62 131 запросов) | Основная нагрузка                                                                         |
| Загрузка `/upload`            | **~9 RPS**                     | Работа с файлами и постановка задач в Celery.                                             |
| `GET /image/{id}` (полный)    | **~7 RPS**                     | Кэшируется 60 с после завершения обработки                                                |
| Fails                         | **19 из 71992**                | Доля отказов 0,026%                                                                       |

```bash
docker compose --profile loadtest up locust

```
